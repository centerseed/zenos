# ZenOS 行銷自動化 — Feature Spec

> 日期：2026-03-20
> 狀態：Draft
> 負責角色：PM（spec）→ Architect（技術設計）→ Developer（實作）

---

## 問題陳述

Barry 需要持續經營 Threads 行銷，但每天手動追蹤市場動態、每週規劃內容主題耗時且容易斷掉。
需要一個自動化流程：每天產出市場調查報告，每週據此產出行銷計劃，形成持續的內容引擎。

## 目標

- 每天自動產出一份台灣市場調查報告（AI 落地 / 中小企業數位轉型 / CRM·ERP 相關）
- 每週自動產出行銷計劃，包含具體的素材 brief（文章、Threads 短文、圖卡）
- 所有報告與素材存入 `campaigns` collection，累積成可查詢的知識庫

## 非目標（不在範圍內）

- 不自動發布到 Threads（Barry 手動發布，保留品質控制）
- 不追蹤 Threads engagement metrics（v1 先靠 Barry 口頭回饋）
- 不做 LinkedIn / Medium / Facebook 等其他平台
- 不做付費廣告投放

## 場景一：每日市場調查

### 場景描述

每天早上，系統自動搜尋網路上與 AI 落地、中小企業數位轉型、CRM/ERP 相關的最新資訊（以台灣中文市場為主），辨識內容缺口與趨勢，產出結構化市場調查報告。

### 觸發

排程觸發，每日一次（建議早上 7:00，Barry 起床前跑完）

### 輸入

| 來源 | 內容 | 格式 |
|------|------|------|
| 網路搜尋 | 台灣市場：AI 落地案例、中小企業轉型動態、CRM/ERP 導入趨勢 | 自然語言 |
| 本地累積 | 過去 7 天的市場報告（避免重複報導） | Markdown |

### 輸出

結構化市場調查報告，包含以下區塊：

```markdown
# 市場調查報告 — YYYY-MM-DD

## 今日重點（3-5 則）
- [標題]：[一句話摘要] | 來源：[URL]

## 趨勢觀察
- [辨識出的趨勢或轉變，跟前幾天的報告做對比]

## 內容缺口（GAPS 分析）
- [競品/市場上沒人講清楚的主題]
- [機會評分：商業影響力 × 競爭難度]

## 素材靈感
- [可以拿來寫 Threads 的切角，2-3 個]

## 原始資料來源
- [URL 列表]
```

### 成功條件

- [ ] 報告在每天 07:30 前自動產出並存檔
- [ ] 報告涵蓋至少 3 則當日或近日的市場資訊
- [ ] 報告的「內容缺口」區塊提供可操作的素材方向
- [ ] 不重複報導過去 7 天已涵蓋的主題（除非有重大更新）

### 異常處理

| 異常情境 | 系統行為 |
|----------|----------|
| 搜尋結果不足（< 3 則相關資訊） | 報告標記「今日市場動態較少」，仍產出趨勢觀察和素材靈感 |
| 排程未觸發 | 下次執行時自動補跑缺失日期 |

## 場景二：每週行銷計劃

### 場景描述

每週一早上，系統讀取過去一週的市場調查報告，結合 Barry 對上週素材的回饋（如有），運用 Content Pillar + Repurposing 方法論，產出該週的行銷計劃。

### 觸發

排程觸發，每週一一次（建議早上 8:00，在每日市場報告之後）

### 輸入

| 來源 | 內容 | 格式 |
|------|------|------|
| 本地累積 | 過去 7 天的市場調查報告 | Markdown |
| Barry 回饋（選用） | 上週素材的表現描述 | 自然語言（手動輸入或口頭記錄） |
| 本地累積 | 過去產出的素材 brief（避免主題重複） | Markdown |

### 輸出

結構化週行銷計劃：

```markdown
# 週行銷計劃 — YYYY 第 WW 週（MM/DD ~ MM/DD）

## 本週主題（Content Pillar）
[一句話描述本週的核心主題，從市場調查的內容缺口中選出]

### 選題理由
- 市場訊號：[哪幾天的報告支撐這個選題]
- 內容缺口：[為什麼現在講這個有機會]
- 商業連結：[跟 ZenOS 產品的關聯]

## 核心長文 Brief
- 標題方向：[2-3 個候選標題]
- 目標受眾：[誰會看]
- 大綱（H2/H3）：[結構]
- 關鍵字：[SEO 用]
- CTA：[讀完要幹嘛]
- 參考資料：[來自哪幾天的市場報告]

## Threads 短文 Brief（5-7 則）
每則包含：
- Hook（前 1-2 行）
- 核心觀點（一則一個重點）
- 語氣：口語化、有立場
- 建議 hashtag
- 發布時間建議（週幾）

## 圖卡 Brief（2-3 組）
每組包含：
- 主題標題
- 每頁概念（3-5 頁）
- 數據/圖表需求
- CTA 尾頁文案

## 上週回顧（如有回饋）
- 表現好的素材：[哪則、為什麼]
- 需調整的方向：[具體建議]
```

### 成功條件

- [ ] 計劃在每週一 08:30 前自動產出
- [ ] 核心主題與當週市場調查報告有明確關聯（不是空想）
- [ ] Threads 短文 brief 的 Hook 具體到可以直接拿去改寫發布
- [ ] 不重複過去 4 週已用過的主題（除非有新角度）

### 異常處理

| 異常情境 | 系統行為 |
|----------|----------|
| 過去一週市場報告不足 3 天 | 擴大搜尋範圍至過去 14 天，計劃標記「資料較少」 |
| 無 Barry 回饋 | 跳過「上週回顧」區塊，其餘正常產出 |

## 資料層對應（campaigns collection）

對應架構 spec 中的 `campaigns` collection：

| 資料 | 存法 | campaigns 欄位 |
|------|------|----------------|
| 每日市場報告 | 每份一筆 document | `type: "assets"`, `name: "市場調查-YYYY-MM-DD"` |
| 每週行銷計劃 | 每份一筆 document | `type: "assets"`, `name: "週行銷計劃-YYYY-WW"` |
| Threads 素材 brief | 每則一筆 document | `type: "assets"`, `name: "threads-brief-YYYY-MM-DD-N"` |

✅ Architect 已確認（見 ADR-001）：
- 新增 `subtype` 欄位：market_research / weekly_plan / thread_brief / article_brief / card_brief
- 全部存 Firestore document 內（單份報告 2-30KB，遠低於 1MB 上限）
- 不做本地 Markdown，直接上 Firestore（Barry 確認 Cloud 環境）

## 技術實作方向（粗估）

Agent 層直接用 **Claude Code CLI**，不走 naru_agent。原因：
- Barry 是唯一用戶，不需要 API 化
- Claude Code 原生有 WebSearch、檔案讀寫
- 排程用 macOS cron 或 launchd 即可

### 需要的能力（粗估）

- [ ] **WebSearch**：搜尋台灣市場資訊 ⚠️ 待 Architect 確認搜尋策略
- [ ] **檔案讀寫**：讀取歷史報告、寫入新報告 — Claude Code 原生支援
- [ ] **排程觸發**：每日 + 每週 cron job ⚠️ 待 Architect 確認 cron 設定
- [ ] **Threads 資料讀取**：爬取 Threads 上的相關內容作為市場資訊來源 ⚠️ 待 Architect 確認可行性

## 成功指標

- **短期（1-2 週）**：每天有報告產出、Barry 覺得報告有用（主觀判斷）
- **中期（1 個月）**：基於報告的 Threads 發文頻率達到每週 5+ 則
- **長期（2-3 個月）**：Threads 追蹤人數成長趨勢可見、有潛在客戶透過 Threads 聯繫

## 開放問題

- ✅ ~~Threads 是否有公開 API？~~ → 有。Meta 官方 Keyword Search API，2,200 次/24h。v1 先用 WebSearch，同步申請 API。見 ADR-001。
- ✅ ~~Claude Code CLI 排程穩定性？~~ → VM crontab + wrapper（3 次重試 + 補跑 + 監控）。見 ADR-001。
- ✅ ~~Barry 回饋如何輸入？~~ → v1：透過 Claude Code 對話輸入，AI 寫入 Firestore。見 ADR-001。
- ✅ ~~品牌語氣指南？~~ → v1 不預先定義，先跑起來再從實際產出中收斂。見 ADR-001。

## 優先級

| 項目 | 優先級 | 理由 |
|------|--------|------|
| 每日市場調查報告 | **P0** | 整個行銷引擎的基礎輸入 |
| 每週行銷計劃 | **P0** | 把報告轉化為可執行的素材方向 |
| Threads 素材 brief 產出 | **P0** | 計劃的核心產出物 |
| Threads 資料爬取 | **P1** | 增強市場調查品質，但 v1 用 WebSearch 也能跑 |
| Barry 回饋閉環 | **P1** | 讓計劃越來越準，但 v1 沒有也能運作 |
| Firestore 寫入 | **P2** | v1 先用本地 Markdown，驗證流程後再上 Firestore |
