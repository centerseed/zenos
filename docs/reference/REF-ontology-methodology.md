# ZenOS Ontology 治理方法論

> 版本：v1.0（2026-03-21 原稿，2026-04-23 frontmatter 標準化）
>
> **Canonical 分工**：本 REF 記錄 ontology **方法論思想**（對話建模、分層判準、迭代收斂、治理哲學）。具體 schema / lifecycle / state machine / error code 以 `SPEC-ontology-architecture v2` + `SPEC-task-governance` + `SPEC-doc-governance` + `SPEC-governance-framework` 為權威。本 REF 中若敘述具體 DDL 或狀態轉換與 SPEC 不一致，以 SPEC 為準。
> 來源：從 ZenOS 自身 + Paceriz ontology 建構實戰中提煉

---

## 這份文件是什麼

這是 ZenOS 建構和治理 ontology 的完整方法論。未來對任何客戶導入 ontology，都照這套流程走。

核心問題只有一個：**面對一間公司的一堆資料，怎麼判斷要建幾份 ontology、每份多深、哪些該拆開、哪些該合併？**

---

## 第一步：判斷 Ontology 的拆分粒度

### 判斷規則：一個「值得獨立的實體」= 一份模組 ontology

一個實體值得獨立成一份 ontology 的條件（滿足任意兩項）：

| 條件 | 判斷方法 | 範例 |
|------|---------|------|
| **有 3+ 份文件圍繞它** | 計算神經層 entry 中關聯到這個實體的文件數 | ACWR：EXPERT_FEEDBACK.md + LITERATURE_REVIEW.md + VDOT.ipynb = 3 份 |
| **有獨立的依賴鏈** | 它跟其他模組之間有明確的輸入/輸出關係 | 數據整合 → 輸出統一數據模型 → ACWR 和 Rizo 都依賴它 |
| **有不同的讀者群** | 關心這個實體的人跟關心其他實體的人不同 | ACWR 的讀者是跑步專家 + 開發者，官網的讀者是行銷 |
| **有獨立的目標或待決策項** | 這個實體有自己的路線圖或未解問題 | 訓練計畫有 v2 重構目標，ACWR 有安全修復目標 |
| **複雜度超過一段話能說清楚** | 如果在 index.md 裡要花超過 5 行解釋，就該獨立 | Rizo AI 的能力、限制、POC 歷史不是一句話能講完的 |

### 不該獨立的實體

| 情境 | 處理方式 |
|------|---------|
| 只有 1 份文件關聯 | 留在 index.md 的全景圖裡，不需要獨立模組 |
| 是另一個模組的子功能 | 寫在母模組的 ontology 裡（如 VDOT 寫在 ACWR 相關裡） |
| 純粹是基礎設施（如 Firestore） | 在 index.md 的技術架構表裡列出即可 |

### 實戰範例：Paceriz 的拆分決策

```
掃描完所有文件後，識別出的實體候選：

  Rizo AI 教練          → 4 份 POC notebook + 核心差異化 + 獨立讀者群     → ✅ 獨立
  訓練計畫系統          → 5 份文件 + v2 重構目標 + 獨立依賴鏈            → ✅ 獨立
  運動數據整合          → 1 份大文件(395行) + 獨立依賴鏈 + 平台擴展目標  → ✅ 獨立
  ACWR 安全機制         → 3 份文件 + 安全修復目標 + 專家讀者群           → ✅ 獨立
  VDOT 跑力計算         → 1 份 notebook + 是 ACWR 的子功能              → ❌ 不獨立，寫在 ACWR 裡
  官方網站              → 1 份 README + 簡單                            → ❌ 不獨立，寫在 index 裡
  用戶留存分析          → 1 份指南 + 工具性質                           → ❌ 不獨立，寫在 index 裡
  Flutter App           → 2 份文件但都是開發規範，不是業務知識           → ❌ 不獨立
  Firestore             → 基礎設施                                     → ❌ 不獨立

結果：4 個模組 ontology + 1 個 index + 1 個盲點分析 + 1 個神經層索引 = 7 份文件
```

### 拆分粒度的規模對照

| 公司規模 | 產品數 | 預估模組數 | Ontology 文件數 |
|---------|--------|-----------|---------------|
| 一人公司（1 個產品） | 1 | 2~4 | 5~7 |
| 小團隊（2~3 個產品） | 2~3 | 5~10 | 10~20 |
| 中型公司（5+ 產品線） | 5+ | 15~30 | 30~50 |

隨公司規模增長，index.md 不會變大（它永遠是 2 分鐘的全景），變大的是 modules/ 的數量和每個模組的深度。

## 第二步：建構流程（完整 SOP）

### Phase A — 盤點（10 分鐘）

**輸入：** 客戶的文件存取權限（Stage 0 = 只有對話，Stage 1 = 部分文件）

**動作：**
1. 列出所有非程式碼文件（.md, .pdf, .docx, .xlsx, 文字 notebook）
2. 不讀內容，只看：檔名、路徑、最後修改時間
3. 產出：文件清單 + 粗略分類（哪些看起來是活的、哪些看起來是歷史記錄）

**判斷標準：**
- 帶日期的檔名（`FIX_2026-02-07.md`）→ 可能是一次性記錄
- 「FINAL」「COMPLETE」「SUMMARY」→ 可能是完成報告
- 「CLAUDE.md」「README.md」「SPEC.md」→ 可能是活文件
- 最後修改超過 6 個月 → 標記為可能過時

### Phase B — 骨架層建構（15 分鐘）

**輸入：** 對話（Stage 0）或對話 + 文件（Stage 1）

**動作：**
1. 識別核心實體：產品、功能模組、目標、角色
2. 建立依賴關係圖
3. 為每個實體填 4D 標籤（What/Why/How/Who）
4. 判斷哪些實體值得獨立成模組（用上面的規則）
5. 產出：`index.md`（骨架層全景）

**4D 標籤的填法：**

| 維度 | AI 能自動填的 | 需要人確認的 |
|------|-------------|-------------|
| What（這是什麼） | ✅ 產品名、功能描述、技術組成 | — |
| Who（誰相關） | ✅ 從文件作者和內容推斷 | 角色的實際職責範圍 |
| Why（為什麼做） | ⚠️ 能推斷表面目標 | 優先級、真正的商業意圖 |
| How（怎麼做、什麼階段） | ⚠️ 能推斷技術實現 | 實際進度、時程、資源分配 |

**核心原則：** What/Who 欄位 AI 先填，Why/How 欄位標記 draft 待確認。不要猜優先級。

### Phase C — 模組 Ontology 建構（每模組 5 分鐘）

**輸入：** 骨架層確定的獨立模組列表

**對每個模組：**
1. 寫一段「這個模組在做什麼」（非技術語言）
2. 列出依賴（輸入什麼、輸出什麼）
3. 列出相關文件（從神經層 entry 中篩選）
4. 列出待決策項目
5. 產出：`modules/{module-name}.md`

### Phase D — 神經層建構（15 分鐘）

**輸入：** Phase A 的文件清單 + Phase B 的骨架層

**對每份文件：**
1. 讀文件前 50 行（或標題 + 第一段），判斷 What
2. 從路徑和內容推斷 Who
3. 嘗試關聯到骨架層的實體
4. 判斷狀態（現行 / 可能過時 / 建議歸檔）
5. 產出：`neural-layer.md`

**分類判斷：**

```
分類決策樹：

  文件是否有持續維護的跡象？（CLAUDE.md、SPEC.md、定期更新的結構文件）
    → 是 → 核心文件

  文件是否承載獨特知識？（專家回饋、學術文獻、設計決策記錄）
    → 是 → 知識文件

  文件是開發過程的一次性記錄？（修復報告、重構摘要、完成確認）
    → 是 → 建議歸檔

  文件是 notebook / POC？
    → 是 → 標記為未索引，建議建立一行描述
```

### Phase E — 盲點分析（10 分鐘）

**輸入：** 骨架層 + 神經層的完整 ontology

**推斷方法：**

| 盲點類型 | 怎麼發現 |
|---------|---------|
| **文件放錯位置** | 文件的 What 標籤跟它所在資料夾名稱不匹配（marketing/ 裡放技術文件） |
| **核心功能文件不足** | 骨架層標記為「核心差異化」但相關文件數量 < 2 |
| **已確認問題未排時程** | 文件中有「已確認問題」「待修復」但沒有對應的 How（時程、優先級） |
| **一次性文件佔比過高** | 建議歸檔的文件數 > 核心文件數的 2 倍 → 知識噪音高 |
| **時間線斷裂** | 文件之間的更新時間有大斷層 → 中間的演進可能沒被記錄 |
| **缺少非技術入口** | 骨架層有「行銷夥伴」角色但沒有任何 Who = 行銷的文件 |
| **目標優先級不明** | 多個活躍目標但沒有明確的排序 |

**盲點分級：**
- 🔴 高：影響安全、影響核心業務、影響關鍵角色能否工作
- 🟡 中：影響效率、增加噪音、有潛在風險
- 🟢 低：觀察中、有改善空間但不緊急

### Phase F — 老闆 Review（5 分鐘）

**輸出給老闆的：** 只有 `index.md` + `blindspots.md`

**老闆看完後的三種反應和對應動作：**

```
反應 1：「大致對，但 X 搞錯了」
  → 修正骨架層相關 entry → 標記 confirmedByUser
  → 級聯檢查：X 的修正會不會影響其他模組？

反應 2：「你漏了 Y」
  → 新增實體到骨架層（draft）
  → 判斷 Y 是否需要獨立模組（用拆分規則）
  → 回到 Phase C 建構模組

反應 3：「盲點 Z 很重要，要處理」
  → 把 Z 轉為待決策項目
  → 寫入對應模組的「待決策」區塊
```

## 第三步：持續治理（Phase G — 永續）

### 觸發規則速查

| 事件 | 骨架層動作 | 神經層動作 | 盲點分析動作 |
|------|----------|----------|------------|
| **新建文件** | 檢查是否暗示新實體 | 建 entry（draft） | — |
| **大幅修改文件** | 檢查是否有新實體 | 重新標籤 | 檢查是否產生矛盾 |
| **刪除文件** | — | 標記 archived | — |
| **對話產生新概念** | 新增實體（draft） | — | — |
| **老闆說「方向變了」** | 更新目標 + 級聯 | 相關 entry 重新分類 | 重新執行盲點分析 |
| **一個月沒動靜** | — | 標記 possibly-stale | 跑過時推斷 |

### 過時推斷邏輯

不靠「文件有沒有更新」，靠跨實體活動度異常：

| 推斷模式 | 邏輯 | 動作 |
|---------|------|------|
| 功能更新但文件沒跟 | 模組 A 的代碼活動高但文件未更新 | 建議 review 文件 |
| 目標完成但沒關閉 | 所有相關任務都 done 但目標還 active | 建議關閉目標 |
| 依賴方更新但被依賴方沒反應 | A 依賴 B，B 更新了但 A 沒動 | 建議檢查 A 是否需要同步 |
| 角色消失 | Who 欄位的某角色三個月沒出現在任何文件 | 建議確認角色是否還在 |

### 拆分 / 合併規則

Ontology 不是建完就固定的。隨專案演進會需要拆分或合併：

**何時拆分：** 一個模組的文件數超過 10 份，或內部出現兩組明顯不同的讀者群。
- 範例：「運動數據整合」如果 Strava / Coros / Polar 都上線了，每個平台可能需要獨立模組。

**何時合併：** 兩個模組的文件開始大量重疊，或其中一個的文件數降到 1 份以下。
- 範例：如果 VDOT 後來有了 3 份獨立文件和自己的路線圖，它應該從 ACWR 裡獨立出來。反之如果 ACWR 的問題都修完了只剩維護，它可以合回訓練計畫裡。

## Ontology 文件的標準結構

### index.md（老闆看的）

```
# {產品名} — Ontology 全景
> confirmedByUser: ❌/✅

## 一句話
## 產品現況（表格）
## 功能模組全景（ASCII 圖 + 連結到 modules/）
## 模組間依賴關係（流程圖）
## 目前活躍目標（表格，帶優先級）
## AI 發現的盲點（摘要 + 連結到 blindspots.md）
## 導航（想知道 X → 去看 Y）
```

### modules/{name}.md（高管/開發者看的）

```
# {模組名} — 模組 Ontology
> confirmedByUser: ❌/✅

## 這個模組在做什麼（非技術語言，3~5 行）
## 現在正在發生什麼（活躍的變更、已知問題）
## 依賴（輸入什麼、輸出什麼）
## 相關文件（表格：文件、用途、讀者）
## 待決策
```

### blindspots.md（老闆看的）

```
# {產品名} — 盲點分析
> confirmedByUser: ❌

## 🔴 高優先級（每個盲點：事實 → 推斷 → 影響 → 建議動作）
## 🟡 中優先級
## 🟢 觀察中
```

### neural-layer.md（AI agent 看的）

```
# {產品名} — 神經層
> confirmedByUser: ❌

## 核心文件（表格：路徑、What、Why、How、Who、關聯模組、狀態）
## 知識文件
## Notebook POC
## 建議歸檔的文件
## 統計
```

## 品質檢查清單

建構完每份 ontology 後，用這個清單自檢：

- [ ] 老闆讀 index.md 是否能在 2 分鐘內理解產品全貌？
- [ ] 依賴關係圖有沒有遺漏的關鍵連結？
- [ ] 每個 ⚠️ 待確認的欄位都有標記嗎？
- [ ] 盲點分析是否從「文件交叉比對」推斷，而不是主觀判斷？
- [ ] 神經層的每份文件都有明確的關聯模組嗎？
- [ ] 建議歸檔的文件有清楚的判斷依據嗎？
- [ ] 目標是否有明確的優先級（或標記「優先級待定」）？
- [ ] 是否有角色（Who）在骨架層出現但在神經層找不到相關文件？
- [ ] 拆分粒度是否合理？（每個模組 3~10 份相關文件）

L2（module）治理硬規則補充：
- [ ] 每個 active L2 都至少有 1 條具體 `impacts`（`A 改了什麼 -> B 的什麼要跟著看`）
- [ ] 新候選 L2 若無法給出具體 `impacts`，不得升為正式 L2（需補 impacts、降級為 L3 或重切粒度）

## 從 Dogfooding 學到的

### ZenOS 自身（簡單案例）

- 20 個實體，1 份 REF-ontology-current-state.md 夠用
- 因為 Phase 0 沒有大量文件，骨架層和神經層可以合在一起
- 隨 Phase 1 展開，會需要按領域拆分（product / technical / business / operations）

### Paceriz（中等案例）

- 45 份文件 → 4 個獨立模組 + 15 份建議歸檔
- 單一 ontology 文件放不下（放了也沒人想讀）
- 拆分後 index.md 控制在 2 分鐘閱讀量，每個模組 1 分鐘
- 盲點分析獨立出來是對的 — 它的讀者（老闆）和更新頻率（持續）跟模組不同
- 神經層獨立是對的 — 它是給 AI 看的路由表，不是給人讀的

### 關鍵教訓

1. **先建骨架再建神經，不要同時做。** 骨架定義了「有哪些實體」，神經層才能把文件關聯到正確的實體。
2. **盲點是最有價值的產出。** 老闆不缺「現狀描述」，缺的是「我沒注意到什麼」。
3. **Why/How 不要猜。** 標 draft 比猜錯好。猜錯的 ontology 比沒有 ontology 更危險。
4. **歸檔建議要有判斷依據。** 不能只說「這份應該歸檔」，要說「因為它帶日期且是一次性修復報告」。
5. **一份 ontology 的上限是 5 分鐘閱讀量。** 超過就該拆。index.md 是 2 分鐘，模組是 1 分鐘。

---

*這份方法論本身會隨著更多客戶案例持續迭代。每次建完一個新客戶的 ontology，都要回來看有沒有新的模式可以提煉。*
