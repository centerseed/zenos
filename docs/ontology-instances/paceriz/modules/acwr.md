# ACWR 安全機制 — 模組 Ontology

> 骨架層實體 | confirmedByUser：❌

## 這個模組在做什麼

ACWR（Acute:Chronic Workload Ratio）= 急性負荷 / 慢性負荷。學術共識：ACWR 在 0.8~1.3 之間受傷風險最低，超過 1.5 風險急升。Paceriz 用 ACWR 作為訓練計畫的安全閘門，確保用戶不會訓練過量。

**設計原則：** 10% 漸進原則（ACWR 上限 1.1），安全優先。

## ⚠️ 已確認的問題（來自兩位跑步專家回饋）

| 問題 | 嚴重性 | 現狀 | 風險 |
|------|--------|------|------|
| **第一週 ACWR = 9999（無保護）** | 🔴 高 | 新用戶沒有歷史數據，ACWR 被設為 9999（不限制） | 用戶可能設定過高初始跑量，直接受傷 |
| **Taper 後恢復邏輯缺失** | 🔴 高 | Taper 結束後沒有特殊處理，可能從低跑量直接跳回高跑量 | 比賽後的恢復期特別脆弱 |
| **Layer 1 邏輯一致性** | 🟡 中 | 絕對上限的文檔描述跟代碼 min() 邏輯有衝突 | 文檔說可以突破上限，但代碼不允許 |

**有爭議的項目（專家意見不一致）：**
- 全馬 180km 絕對上限是否過高？（一說過高，一說留給精英）
- ACWR 1.1 上限是否過保守？（一說合理，一說應 1.2-1.3）
- 長跑比例上限 30%？（兩位專家建議相反）

## 依賴

```
歷史運動數據 → 計算慢性負荷（過去 4 週平均）
當週訓練數據 → 計算急性負荷
→ 輸出：安全跑量上限 → 給訓練計畫系統使用
```
