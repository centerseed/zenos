# /brainstorm — 系統性腦暴流程

用多視角挑戰避免過早收斂，讓想法經過壓力測試後更健壯。

---

## 觸發條件

用戶說「brainstorm」「腦暴」「多想幾個角度」「挑戰一下」「幫我找盲點」時啟動。

---

## 輸入

需要用戶提供或從對話上下文取得：
- **當前分析/想法/決策草案**（必須有具體內容，不能是空的）

如果用戶只說「brainstorm」但沒給內容，詢問：「你想挑戰的想法或分析是什麼？」

---

## Phase 1：擴散 — Challenger 挑戰

叫起 Challenger agent（讀取 `skills/release/challenger/SKILL.md`），傳入當前分析內容。

Challenger 執行完整流程：
1. Steel-man 重述原始觀點
2. 動態選擇 3-5 個最相關的挑戰視角
3. 產出結構化的挑戰報告

**Gate**：Challenger 報告產出後，展示給用戶，詢問：
- 「這些挑戰中，哪些最讓你覺得需要回應？有沒有想補充的角度？」

---

## Phase 2：收斂 — 回應與整合

根據用戶選擇的重點挑戰，回到原始分析的角色（或用戶自己），逐一回應：

對每個被選中的挑戰：
```
### 挑戰 #{n}：{挑戰核心}

回應：
- 接受 / 部分接受 / 反駁
- {具體論證}

對原始結論的調整：
- {如果接受，怎麼修改}
- {如果反駁，補充什麼論據}
```

---

## Phase 3：升級版結論

產出經過挑戰後的升級版分析：

```
## 升級版結論

### 原始主張（調整後）
{經過挑戰修正的版本}

### 新增的條件與限制
{原本沒考慮到但現在加入的}

### 仍然存在的風險
{挑戰中無法完全回應的，標註為已知風險}

### 值得進一步探索的方向
{挑戰過程中浮現的新可能性}
```

---

## Phase 4：寫入 Work Journal（只在重大結論時）

只有當 brainstorm 產生會影響後續產品/技術方向的重大結論時，才記錄工作摘要。一般討論不要寫 journal。

```python
mcp__zenos__journal_write(
    summary="brainstorm {主題}：{升級版結論的一句話摘要}",
    project="{專案名}",
    flow_type="research",
    tags=["{主題關鍵字}"]
)
```

---

## 注意事項

- Phase 1 和 Phase 2 之間**必須有用戶確認**，不能自動跑完
- Challenger 不給結論，收斂階段才整合
- 如果挑戰揭示了根本性問題（核心假設不成立），直接告知用戶，不要硬收斂
