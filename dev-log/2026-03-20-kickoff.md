## 產品定位

**ZenOS** 是一套 AI-Native 的小型公司作業系統，定位為 Barry 的**落地顧問產品**。

開發策略分兩個階段：
1. **先為自己用**：在自己的業務中驗證架構與流程
2. **再部署給客戶**：以顧問導入的方式水平拓展出去

核心價值主張：
> 「不是再裝一套 Odoo。是讓你們現有的資料開始會說話，員工不需要改習慣。」

---

## 架構共識

### 四大業務種類（公司治理角度）

```
行銷          獲客、市場、素材           → campaigns
管理          金流、開銷、收支           → finances
開發          需求、方向、進度           → projects
客服 / CRM    客戶、合作夥伴            → contacts、orders
```

### 三層結構

```
展示層        儀表板、批次操作、通知規則管理、權限設定、報表分析
Agent 層      四個 Agent：輸入 / 查詢 / 監控 / 簽核
              AI 核心能力：排程自動化 / 責任路由 / 信任曲線管理 / 知識沉澱
資料層        Firestore 五大 Collection：contacts / finances / projects / campaigns / orders
              共用原則：sourceText 必存、confirmedByUser 防污染、aliases[] 同義對應、null 優於猜測
```

### Agent 層的關鍵決策：可抽換設計

AI 引擎與業務邏輯解耦，介面一致，引擎可替換：

```
業務邏輯層（MCP Tools + Prompts）
        │
        ├── Claude Agent SDK（訂閱制，高能力）
        └── naru_agent + Anthropic API（按量付費，輕量）
```

**MCP Tools 是兩個引擎都吃的格式**，工具定義寫一次，引擎隨便換。

## 技術棧決策

### AI 引擎選型

| 情境 | 引擎 | 原因 |
|------|------|------|
| Barry 自己開發 / 驗證 | Claude Code CLI | 最高能力，訂閱費攤平 |
| 認真導入的客戶 | Claude Agent SDK | 訂閱制，Claude Code 引擎直接暴露為 SDK |
| 輕量客戶 / 試用期 | naru_agent + API | 按量付費，門檻低 |

### Claude Agent SDK vs naru_agent

- **Claude Agent SDK**：Claude Code 的 agentic loop 直接程式化，Claude 自己決定工具呼叫順序，原生 MCP 支援、session 管理、17 種 hooks
- **naru_agent**：明確的排程控制（AgentPipeline / AgentFanout / AgentHandoffLoop），適合需要精確流程控制的場景

兩者互補，不互斥。

### 部署模型：BYOS（Bring Your Own Subscription）

- 每個客戶 = 一個 VM + 一個 Claude 訂閱帳號
- Barry 提供打包好的環境 + 顧問導入服務
- 客戶自付 Anthropic 費用，Barry 收顧問費
- **ToS 合規**：每個客戶用自己的帳號

成本比較：
- Claude Max（$200/month）≈ API 等效用量的 10 倍以上 token
- 小型客戶每月 API 用量估計不超過 $10–50

### 參考資源

- **AgentAPI by Coder**（1.1k ⭐）：把 Claude Code CLI 包成 HTTP API + SSE streaming
- **Claude Agent SDK**（官方）：`@anthropic-ai/claude-agent-sdk` / `claude-agent-sdk`（Python）
- **naru_agent**（Barry 自建）：已有 AgentOrchestrator、ChannelAdapter、RedisSessionStore 等完整模組

## 現有資源盤點

### naru_agent（已完成）

Barry 已有一套自建的 agent framework，Python + TypeScript 雙版本，包含：
- `NaruAgent`：核心 agent
- `AgentOrchestrator`：4 階段路由（Pending / Intent / Direct / Delegate）
- `AgentPipeline / AgentFanout / AgentHandoffLoop`：可組合 primitives
- `ChannelAdapter`：LINE / Slack / API 接入抽象
- `RedisSessionStore`：stateless 多 instance 部署

ZenOS 的 Agent 層直接建在 naru_agent 之上。

## 開發策略

**從應用面回推，不從底層往上蓋。**

原因：
- 避免過度設計底層 Schema
- 讓真實使用場景驅動工具定義
- 盡快跑通一個完整流程，建立信任曲線

每個場景倒推出需要的 Schema 和 MCP Tools，自然收斂出完整底層。
